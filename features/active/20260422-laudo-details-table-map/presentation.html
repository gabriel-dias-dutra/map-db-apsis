LaudoDetails Table Map

A focused, trustworthy map of the LaudoDetails table — every column, every direct relationship, every data-shape trap — so the n8n search agent builds correct SQL about laudos.

Created on April 22, 2026

Feature overview

The single piece of structural knowledge the search agent is missing about the laudos domain — narrowly scoped, self-contained, ready to paste into the agent's system prompt.

The Apsis team already has a database search agent running inside an n8n workflow that is meant to turn Portuguese business questions about laudos into correct SQL. Today the agent fails because it does not know the shape of LaudoDetails — its columns, their meaning, the tables it relates to, or the fact that many numeric and date fields are stored as text. This feature delivers a single committed Portuguese map of that one table, generated by a small script that can be rerun whenever the table changes, and pasted directly into the agent's prompt.

Feature goals

Focused, measurable outcomes for this feature.

  • Give the search agent a complete, trustworthy understanding of LaudoDetails so it builds correct SQL — including correct JOINs to neighboring tables.
  • Prevent the most likely silent failure: the agent filtering, sorting, or aggregating text-stored columns as if they were numbers or dates.
  • Keep the effort proportional to the scope — one table and its direct neighbors — while leaving the script structured so mapping another table later is inexpensive.
  • Make regeneration cheap enough that the map stays current as the table evolves — target: a full refresh in under five minutes.

How it works

Three small steps, executed manually on demand by any Softo developer.

Schema introspection

A single command inside repos/sql/ walks the SQL Server catalog through the existing Lambda, capturing columns, foreign keys, extended-property descriptions, row counts, and name-inferred relationships. It writes a clean JSON snapshot that the next step consumes.

Map generation

The snapshot is rendered into a Portuguese Markdown map. It opens with a prominent "Avisos de tipo" section listing every text-stored numeric or date column, repeats the flag inline next to each affected column, and documents relationships labeled by discovery source (FK, inferred, or by value).

Agent integration

The developer replaces the laudos section of the n8n agent's system prompt — delimited by clear marker comments — with the generated Markdown. A short README section in the sql repo documents the commands and their outputs.

Future ideas

Ideas that came up during planning and may fit later iterations.

  • Parameterize by table name — generalize map-laudo-details into map-table once a second table is requested, with per-table outputs side by side.

  • Validation question battery — 5–10 committed laudos questions with expected answers, so regressions are caught objectively instead of anecdotally.

  • Apsis semantic review — ask the Apsis team to review financial term descriptions (UnleveredBeta, GrossMarginPerpetuity, GICS) and replace them with authoritative language.

  • Extended-property authoring — push confirmed descriptions back into SQL Server sys.extended_properties, making the database itself the source of truth.

  • Stale-map indicator — embed a hash of the table's DDL in the map so the agent (or a CI check) can warn when the live schema no longer matches.

  • Second-hop context on demand — if the agent needs context beyond direct neighbors, add a lookup tool that fetches per-table fact sheets without bloating the system prompt.

Additional notes

Context and constraints worth keeping visible.

  • No new infrastructure. All work lives inside repos/sql/; the existing SQL Lambda (Node.js, mssql v11, x-api-key, 5-second connection timeout) is the only transport.
  • Credentials live in .env.tool at the project root (URL, API_KEY).
  • No declared PK or FKs in the provided CREATE TABLE. The script handles both cases and labels every discovered relationship by discovery mechanism so the reader can judge trust.
  • Softo authors all semantics, drawing on general domain knowledge. Columns that cannot be confidently described are marked "pendente" with a short reason and completed in a follow-up.
  • The archived feature 20260422-db-schema-map-for-llm-agent is the previous whole-database approach. Nothing from it is being reused directly.
