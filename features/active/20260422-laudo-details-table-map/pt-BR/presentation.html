Mapa da tabela LaudoDetails

Um mapa focado e confiável da tabela LaudoDetails — cada coluna, cada relacionamento direto, cada armadilha de forma de dado — para o agente de busca no n8n construir SQL correto sobre laudos.

Criado em 22 de abril de 2026

Visão geral

A peça isolada de conhecimento estrutural que falta ao agente sobre o domínio de laudos — com escopo estreito, autossuficiente, pronta para ser colada no system prompt.

A Apsis já tem um agente de busca rodando num workflow n8n, desenhado para transformar perguntas de negócio em português sobre laudos em SQL correto. Hoje o agente falha porque não conhece a forma da tabela LaudoDetails — suas colunas, o significado delas, as tabelas com as quais se relaciona, nem o fato de que vários campos numéricos e de data estão guardados como texto. Esta feature entrega um único mapa commitado, em português, dessa única tabela, gerado por um pequeno script que pode ser re-executado sempre que a tabela mudar, e colado diretamente no prompt do agente.

Objetivos da feature

Resultados focados e mensuráveis para esta entrega.

  • Dar ao agente de busca uma compreensão completa e confiável de LaudoDetails para que ele construa SQL correto — inclusive JOINs corretos com tabelas vizinhas.
  • Evitar a falha silenciosa mais provável: o agente filtrando, ordenando ou agregando colunas guardadas como texto como se fossem números ou datas.
  • Manter o esforço proporcional ao escopo — uma tabela e seus vizinhos diretos — deixando o script estruturado de modo que mapear outra tabela depois seja barato.
  • Tornar a regeneração barata o bastante para o mapa se manter atual conforme a tabela evolui — meta: refresh completo em menos de cinco minutos.

Como funciona

Três passos pequenos, executados manualmente sob demanda por qualquer desenvolvedor Softo.

Introspecção do schema

Um único comando dentro de repos/sql/ percorre o catálogo do SQL Server através do Lambda existente e captura colunas, foreign keys, descrições de extended properties, contagens de linhas e relacionamentos inferidos por nome. Grava um snapshot JSON limpo que o próximo passo consome.

Geração do mapa

O snapshot é renderizado num Markdown em português. Abre com uma seção proeminente de "Avisos de tipo" listando toda coluna numérica ou de data guardada como texto, repete o aviso inline ao lado de cada coluna afetada e documenta relacionamentos rotulados pela origem da descoberta (FK, inferido ou por valor).

Integração com o agente

O desenvolvedor substitui a seção de laudos do system prompt do agente n8n — delimitada por comentários-marcador claros — pelo Markdown gerado. Uma seção curta de README no repositório sql documenta os comandos e suas saídas.

Métricas de sucesso

Como o escopo é uma única tabela, o sucesso é medido por cobertura e correção nessa tabela, em vez de por uma bateria ampla de validação.

Métrica Alvo Como medir
Cobertura de colunas 19/19 colunas documentadas ou explicitamente marcadas como "pendente" Checklist no pull request; o próprio gerador imprime a contagem

Ideias futuras

Ideias que surgiram durante o planejamento e podem caber em iterações futuras.

  • Parametrizar o script por nome de tabela — generalizar map-laudo-details em map-table assim que uma segunda tabela for pedida, com saídas por tabela lado a lado.

  • Bateria de perguntas de validação — 5–10 perguntas commitadas com respostas esperadas, para regressões serem detectadas de forma objetiva em vez de anedótica.

  • Revisão semântica da Apsis — pedir ao time da Apsis para revisar as descrições de termos financeiros (UnleveredBeta, GrossMarginPerpetuity, GICS) e substituí-las por linguagem autorizada.

  • Autoria por extended property — empurrar descrições confirmadas de volta para sys.extended_properties do SQL Server, fazendo do próprio banco a fonte da verdade.

  • Indicador de mapa desatualizado — embutir um hash do DDL da tabela no mapa para que o agente (ou um check de CI) avise quando o schema ao vivo não bater mais com o mapa.

  • Contexto de 2 hops sob demanda — se o agente precisar de contexto além dos vizinhos diretos, adicionar uma ferramenta de lookup que traga fact sheets por tabela sem inchar o system prompt.

Notas adicionais

Contexto e restrições que vale manter visíveis.

  • Sem nova infraestrutura. Todo o trabalho vive dentro de repos/sql/; o Lambda de SQL existente (Node.js, mssql v11, x-api-key, timeout de conexão de 5 segundos) é o único transporte.
  • As credenciais vivem em .env.tool na raiz do projeto (URL, API_KEY).
  • Sem PK ou FKs declaradas no CREATE TABLE fornecido. O script trata os dois cenários e rotula cada relacionamento descoberto pelo mecanismo de descoberta, para que o leitor sempre saiba quão confiável ele é.
  • A Softo autora toda a semântica, com base em conhecimento geral de domínio. Colunas que não puderem ser descritas com segurança são marcadas como "pendente" com um motivo curto e completadas em iteração seguinte.
  • A feature arquivada 20260422-db-schema-map-for-llm-agent é a abordagem anterior, de banco inteiro. Nada dela está sendo reaproveitado diretamente.
