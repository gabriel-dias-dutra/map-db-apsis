Give the agent the structural knowledge it needs to turn business questions into correct SQL against a 202-table database.

Feature overview

What this feature delivers and why it matters.

Apsis has a SQL Server database with around 202 tables. A database search agent running inside an n8n workflow is meant to answer business questions — "who is on the RH team?", "show me the laudos tied to Petrobras" — by translating each question into SQL, running it, and summarizing the result. Today the agent has the execution channel (an existing Lambda that runs arbitrary SQL from inside the database's VPC) but no structural knowledge of the data: it does not know which tables exist, what they mean, or how they relate. This feature produces the knowledge layer the agent is missing — a set of artifacts and a generation process that extract the database structure, translate it into human-readable business terms, and feed it to the agent so it can reason about what to query.

Feature goals

What the feature is aiming to achieve.

  • Give the search agent a complete, trustworthy map of the Apsis database so it can build correct SQL against tables it has never seen before.
  • Keep the update process simple: on-demand regeneration by a Softo developer, no scheduled jobs, no new infrastructure in the first iteration.
  • Establish a validation baseline (a fixed set of real questions) so agent improvements can be measured objectively over time.
  • Avoid adding operational complexity (no vector store, no new services) until real usage proves they are needed.

How it works

The deliverables that make up the feature, in the order they come into play.

Schema Extraction

A script, run on demand by a Softo developer, connects to the existing SQL Lambda and runs introspection queries against the SQL Server catalogs to capture all tables, columns, data types, relationships and existing business descriptions.

Database Map

A Markdown document in Brazilian Portuguese that lists the 202 tables grouped by business domain, with one-line descriptions and the main relationships — loaded directly into the agent's system prompt.

Validation Question Battery

Around 20 to 30 realistic questions drafted by Softo and validated by Apsis. The agent's hit rate on this battery becomes the baseline for measuring answer quality over time.

Operator Guide

Short operational documentation covering how to run the extraction, review the map, and update the agent's system prompt in n8n — so anyone on the Softo team can refresh the map.

Table Fact Sheets

One detailed sheet per table with columns, relationships, sample rows and example queries. The agent pulls a sheet by exact table name on demand, only when it needs more detail than the map provides.

Table Lookup Mechanism

A simple channel — either an additional endpoint on the existing Lambda or direct HTTP access to the files — so the agent can fetch a fact sheet by name in a single call.

Agent Integration

The n8n workflow is extended so the agent exposes the lookup as a tool, pulling fact sheets in runtime when the user's question requires table-level detail.

Future ideas

Ideas for later iterations, triggered by real usage.

  • Real-question corpus (Q&A layer) — once the agent runs in production, collect real question → SQL pairs as a semantic retrieval layer that sharpens accuracy on recurring questions.

  • Semantic search backed by OpenSearch — when the Q&A corpus grows beyond deterministic lookup, move it to the OpenSearch cluster already available in the stack.

  • Views and stored procedures — extend the extraction once their presence and relevance in the Apsis database are confirmed during use.

  • Automated regeneration — move from manual execution to a trigger (schema-change detection or cron) so the map is never silently stale.

  • Stale-map warning — embed a schema hash or timestamp in the map so the agent can flag when it may be older than the live database.

  • Cardinality annotations — add row counts and value distribution per table so the agent can choose smarter filtering strategies.

  • Exploration side-effects — the first regeneration doubles as a discovery pass; whatever surfaces (views, PII, unusual conventions) may reshape the backlog.

Additional notes

Context, constraints and assumptions worth keeping visible.

  • Lambda reused on both sides: the existing Lambda in repos/sql/ serves both the agent's runtime queries and the introspection queries from the extraction script — no new infrastructure for Milestone 1.
  • Credentials in place: the Lambda's API credentials are already available to the project in .env.tool.
  • 5-second connection timeout: extraction queries must be split into small batches to stay under the Lambda's database connection timeout.
  • PII assumption: PII is assumed absent from the Apsis database. If exploration shows otherwise, the plan for sample data in Milestone 2 is revisited.
