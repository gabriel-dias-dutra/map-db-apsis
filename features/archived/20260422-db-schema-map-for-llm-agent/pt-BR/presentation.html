Mapa de Schema SQL Server para Agente de Busca em Banco

Dar ao agente o conhecimento estrutural necessário para transformar perguntas de negócio em SQL correto contra um banco de 202 tabelas.

Criado em 22/04/2026

Visão geral da feature

O que esta feature entrega e por que importa.

A Apsis tem um banco SQL Server com cerca de 202 tabelas. Um agente de busca em banco rodando dentro de um workflow n8n deve responder perguntas de negócio — "quem faz parte do time de RH?", "informações sobre laudos da Petrobras" — traduzindo cada pergunta em SQL, executando e resumindo o resultado. Hoje o agente tem o canal de execução (uma Lambda existente que roda SQL arbitrário de dentro da VPC do banco), mas não tem conhecimento estrutural dos dados: não sabe quais tabelas existem, o que significam nem como se relacionam. Esta feature produz a camada de conhecimento que falta ao agente — um conjunto de artefatos e um processo de geração que extraem a estrutura do banco, traduzem para termos de negócio e alimentam o agente para que ele possa raciocinar sobre o que consultar.

Objetivos da feature

O que a feature pretende atingir.

  • Dar ao agente de busca um mapa completo e confiável do banco da Apsis para que ele monte SQL correto contra tabelas que nunca viu antes.
  • Manter o processo de atualização simples: regeneração sob demanda por um desenvolvedor da Softo, sem jobs agendados, sem nova infraestrutura na primeira iteração.
  • Estabelecer uma base de validação (um conjunto fixo de perguntas reais) para que melhorias futuras no agente possam ser medidas objetivamente ao longo do tempo.
  • Evitar adicionar complexidade operacional (sem vector store, sem novos serviços) até que o uso real prove necessidade.

Como funciona

Os entregáveis que compõem a feature, na ordem em que entram em ação.

Extração do Schema

Um script, rodado sob demanda por um desenvolvedor da Softo, conecta-se à Lambda SQL existente e executa queries de introspecção contra os catálogos do SQL Server para capturar todas as tabelas, colunas, tipos, relacionamentos e descrições de negócio já existentes.

Mapa do Banco

Um documento Markdown em português brasileiro que lista as 202 tabelas agrupadas por domínio de negócio, com descrições de uma linha e os principais relacionamentos — carregado diretamente no system prompt do agente.

Bateria de Perguntas de Validação

Cerca de 20 a 30 perguntas realistas rascunhadas pela Softo e validadas pela Apsis. A taxa de acerto do agente nessa bateria vira a base para medir qualidade de resposta ao longo do tempo.

Guia do Operador

Documentação operacional curta cobrindo como rodar a extração, revisar o mapa e atualizar o system prompt do agente no n8n — para que qualquer pessoa do time Softo consiga atualizar o mapa.

Fichas por Tabela

Uma ficha detalhada por tabela com colunas, relacionamentos, linhas de exemplo e queries de referência. O agente puxa a ficha pelo nome exato da tabela sob demanda, apenas quando precisa de detalhe além do mapa.

Mecanismo de Lookup por Tabela

Um canal simples — seja um endpoint adicional na Lambda existente ou acesso HTTP direto aos arquivos — para o agente buscar uma ficha pelo nome em uma única chamada.

Integração com o Agente

O workflow no n8n é estendido para que o agente exponha o lookup como tool, puxando fichas em runtime quando a pergunta do usuário exige detalhe em nível de tabela.

Ideias futuras

Ideias para iterações seguintes, disparadas pelo uso real.

  • Corpus de perguntas reais (camada Q&A) — assim que o agente começar a rodar em produção, coletar pares reais pergunta → SQL como uma camada de busca semântica que melhora a precisão em perguntas recorrentes.

  • Busca semântica com OpenSearch — quando o corpus Q&A crescer além do lookup determinístico, movê-lo para o cluster OpenSearch já disponível na stack.

  • Views e stored procedures — estender a extração assim que a presença e relevância delas no banco da Apsis forem confirmadas durante o uso.

  • Regeneração automática — sair da execução manual para um gatilho (detecção de mudança de schema ou cron) para que o mapa nunca fique silenciosamente desatualizado.

  • Aviso de mapa desatualizado — embutir um hash de schema ou timestamp no mapa para que o agente sinalize quando ele pode estar mais antigo que o banco ao vivo.

  • Anotações de cardinalidade — adicionar contagens de linhas e distribuição de valores por tabela para que o agente escolha estratégias de filtragem mais inteligentes.

  • Efeitos colaterais da exploração — a primeira regeneração também serve como passada de descoberta; o que aparecer (views, PII, convenções incomuns) pode remodelar o backlog.

Notas adicionais

Contexto, restrições e premissas que merecem ficar visíveis.

  • Lambda reutilizada dos dois lados: a Lambda existente em repos/sql/ atende tanto as queries de runtime do agente quanto as queries de introspecção do script de extração — nenhuma nova infraestrutura para o Milestone 1.
  • Credenciais disponíveis: as credenciais de API da Lambda já estão disponíveis ao projeto em .env.tool.
  • Timeout de conexão de 5 segundos: queries de extração precisam ser fatiadas em lotes pequenos para caber dentro do timeout de conexão do banco pela Lambda.
  • Premissa de PII: assume-se que não há PII no banco da Apsis. Se a exploração provar o contrário, o plano para sample data no Milestone 2 é revisitado.
